# ADR-0008: 학교 이메일 인증 코드는 MySQL에 해시로 저장한다

## 상태

Accepted

## 날짜

2026-05-14

## 맥락

UniT는 학교 이메일 인증을 통해 대학생 회원의 학교 소속을 확인한다. 인증 코드는 짧은 시간만 유효한 데이터이므로 Redis에 저장하는 방식이 일반적이다.

하지만 Sprint 1에서는 인프라를 MySQL 중심으로 시작하고 Redis 도입은 지표와 필요성이 확인된 뒤 결정하기로 했다.

인증 코드 원문은 사용자에게 이메일로 전달되지만, 서버 저장소에 원문으로 남기면 유출 시 악용될 수 있다.

## 결정

학교 이메일 인증 코드는 MySQL 테이블에 저장하되, 코드 원문은 저장하지 않고 해시만 저장한다.

인증 코드 레코드는 회원 ID, 학교 ID, 이메일 해시, 코드 해시, 목적, 상태, 시도 횟수, 만료 시각을 가진다. 검증 시 사용자가 제출한 코드를 해시하여 저장된 해시와 비교한다.

## 이유

- Redis 없이도 Sprint 1의 학교 인증 기능을 완성할 수 있다.
- 코드 원문을 저장하지 않아 인증 코드 유출 위험을 줄인다.
- 상태, 시도 횟수, 만료 시각을 DB에 남겨 테스트와 운영 확인이 쉽다.
- MySQL 기반으로 시작하면 로컬 개발 환경과 CI 구성이 단순하다.

## 결과

- 인증 코드 조회와 갱신이 DB I/O를 사용한다.
- 만료된 코드, 실패 횟수 증가 같은 상태 변경은 트랜잭션 롤백에 주의해야 한다.
- 인증 코드 만료 처리는 DB 조회 조건과 상태 변경 로직이 함께 필요하다.
- 인증 요청량이 증가하면 Redis 또는 별도 TTL 저장소 도입을 다시 검토한다.

## 대안

- Redis 저장: TTL과 빠른 조회에 유리하지만 현재 단계에서는 인프라 복잡도가 증가한다.
- 인증 코드 원문 저장: 구현은 쉽지만 보안상 선택하지 않는다.
- 이메일 인증 링크 방식: 사용자 경험은 좋을 수 있으나 모바일 앱과 API 중심 흐름에서는 코드 입력 방식이 더 단순하다.
