# ADR-0009: 실패와 만료 상태는 비즈니스 예외 롤백과 분리해 저장한다

## 상태

Accepted

## 날짜

2026-05-14

## 맥락

일부 비즈니스 로직은 실패를 응답으로 반환하면서도 실패 사실 자체는 저장해야 한다. 예를 들어 학교 이메일 인증 코드가 틀리면 시도 횟수를 증가시켜야 하고, 인증 코드가 만료되었으면 상태를 `EXPIRED`로 남겨야 한다.

하지만 Spring의 기본 트랜잭션은 RuntimeException이 발생하면 전체 트랜잭션을 롤백한다. `BusinessException`을 던지기 전에 상태를 변경해도 같은 트랜잭션 안이라면 변경 사항이 저장되지 않는다.

이 문제는 Refresh Token 만료 처리, 인증 코드 만료 처리, 인증 실패 횟수 증가처럼 실패 자체가 도메인 상태인 경우에 반복적으로 발생할 수 있다.

## 결정

비즈니스 예외를 던져도 보존해야 하는 실패, 만료, 폐기 상태는 예외 롤백과 분리된 저장 경로를 사용한다.

상태 변경의 성격에 따라 `noRollbackFor`, 별도 recorder 서비스, 또는 `REQUIRES_NEW` 트랜잭션을 사용한다. 학교 이메일 인증 실패와 만료 기록처럼 실패 이력을 반드시 남겨야 하는 경우에는 recorder를 두어 독립적으로 저장한다.

## 이유

- 실패 횟수와 만료 상태는 도메인 감사와 운영 판단의 근거가 된다.
- 예외 응답 때문에 상태 변경이 롤백되면 동일 레코드가 반복 처리된다.
- 나중에 시도 제한, 보안 모니터링, 운영자 확인 기능을 붙일 때 저장된 실패 이력이 필요하다.
- 성공 흐름과 실패 기록 흐름을 분리하면 트랜잭션 의도가 명확해진다.

## 결과

- 실패 처리 코드에서 트랜잭션 경계를 더 신중하게 설계해야 한다.
- recorder 방식은 클래스가 늘어나지만 실패 상태 저장 책임이 분명해진다.
- `noRollbackFor`는 간단하지만 메서드 전체 변경이 커밋될 수 있으므로 적용 범위를 조심해야 한다.
- `REQUIRES_NEW`는 확실하지만 트랜잭션 분리가 많아지면 흐름 이해가 어려워질 수 있다.

## 대안

- 같은 트랜잭션 안에서 상태 변경 후 예외를 던진다: 기본 롤백으로 상태가 저장되지 않아 선택하지 않는다.
- 모든 비즈니스 예외를 롤백하지 않는다: 의도하지 않은 변경까지 커밋될 수 있어 위험하다.
- 실패 상태를 저장하지 않는다: 구현은 단순하지만 보안과 운영 관점에서 근거 데이터가 사라진다.
