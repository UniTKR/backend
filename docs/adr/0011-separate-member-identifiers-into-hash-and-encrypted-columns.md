# ADR-0011: 회원 식별 정보는 해시 컬럼과 암호화 컬럼으로 분리한다

## 상태

Accepted

## 날짜

2026-05-14

## 맥락

회원 이메일과 전화번호는 로그인, 중복 가입 확인, 학교 인증, 알림 발송 등에 사용될 수 있는 민감 정보다.

이 값들을 원문으로 저장하면 조회와 발송은 쉽지만 DB 유출 시 개인정보 노출 위험이 크다. 반대로 단방향 해시만 저장하면 중복 확인과 조회는 가능하지만 이메일 발송처럼 원문이 필요한 기능을 수행할 수 없다.

따라서 조회용 식별자와 복호화가 필요한 데이터를 분리해야 한다.

## 결정

회원 식별 정보는 목적에 따라 해시 컬럼과 암호화 컬럼으로 분리한다.

이메일은 중복 확인과 로그인 조회를 위해 `email_hash`를 사용하고, 필요한 경우에만 발송 목적의 `email_encrypted`를 사용한다. 전화번호도 중복 확인이나 제재 정책을 고려해 `phone_hash`를 둔다.

Refresh Token과 인증 코드는 원문 복원이 필요 없으므로 해시만 저장한다.

## 이유

- 중복 확인과 조회는 단방향 해시만으로 처리할 수 있다.
- 이메일 발송처럼 원문이 필요한 기능은 암호화 컬럼으로 제한한다.
- 토큰과 인증 코드는 원문 복원이 필요 없으므로 원문 또는 양방향 암호화 저장을 피한다.
- 개인정보 보관 목적을 컬럼 단위로 분리하면 탈퇴와 마스킹 정책을 세우기 쉽다.

## 결과

- 로그인과 중복 확인은 입력값을 정규화한 뒤 해시하여 비교한다.
- 이메일 발송이 필요한 흐름은 암호화 컬럼 관리가 필요하다.
- 탈퇴 시 복호화 가능한 개인정보는 제거하고, 운영상 필요한 해시는 정책에 따라 보존할 수 있다.
- 해시 정규화 규칙이 바뀌면 기존 데이터와 호환성 문제가 생길 수 있으므로 초기 규칙을 유지해야 한다.

## 대안

- 이메일 원문 저장: 구현은 쉽지만 개인정보 노출 위험이 커 선택하지 않는다.
- 해시만 저장: 보안성은 높지만 메일 발송 등 원문이 필요한 기능과 충돌한다.
- 모든 개인정보를 외부 KMS 기반 암호화 저장소로 분리: 장기적으로 좋지만 MVP 단계에서는 범위가 크다.
